## Critical Findings

### 1. Randomized Delay Never Invoked (CRITICAL)

**Location:** `src/autonomedia/core/posting_routine.py` line 186-218

**Evidence:**
```bash
$ grep -c '_apply_randomized_delay' src/autonomedia/core/posting_routine.py
1  # Only 1 occurrence - the function definition, never a call
```

**Root Cause:** The `_apply_randomized_delay()` function is defined at line 45 but never invoked in `posting_routine()`. The spec explicitly requires:
> "Apply randomized 2-10 minute delay between platform batches"

**Impact:** No natural pacing between posts. Content is posted immediately without the human-like timing patterns specified.

### 2. Missing TC9/TC10 Tests (HIGH)

**Location:** `tests/test_m14_routine.py`

**Evidence:** The tests `test_first_execution_posts_one_item` and `test_posts_one_to_two_items` do NOT set up `post_history` entries with timestamps to validate the 8-hour window logic.

**Expected Behavior (per Verification):**
- TC9: Second execution within 8 hours → no additional items posted
- TC10: Execution after 8+ hours → second item may be posted

**Actual:** Both tests run with an empty `post_history` table, making `last_posted = None`, which triggers the "post only 1 item" path. This does not validate the 8-hour window.

### 3. Missing TC3 Test (HIGH)

**Location:** `tests/test_m14_routine.py`

**Evidence:** No test for mixed verified/unverified platforms exists. The verification document specifies:
> TC3: Mixed verified/unverified platforms → only verified platforms processed

### 4. Legacy PostingWorker Still Present (MEDIUM)

**Location:** `src/autonomedia/apps/worker/posting_executor.py`

**Evidence:** The `PostingWorker` class exists and queries `ready_to_post` without verification status checks:
```python
rows = await conn.fetch("SELECT * FROM content WHERE status = 'ready_to_post'")
# ... processes without checking verification_status
```

**Impact:** If both workers run simultaneously, there's potential for duplicate posting of unverified content. The spec states this is out of scope, but the legacy code creates an operational hazard.

### 5. Test Runtime Warnings (LOW)

**Evidence:** Test run shows warnings:
```
RuntimeWarning: coroutine 'AsyncMockMixin._execute_mock_call' was never awaited
```

This suggests the mock patches may not be properly awaiting coroutines in some test scenarios.
