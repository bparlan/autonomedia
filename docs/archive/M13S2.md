# Spec: M13S2 - Backend Query Modifications

## Problem

Backend endpoints need to expose per-platform content for the Ready to Publish section and implement platform health filtering for content generation. The current `/rewrites` endpoint excludes `ready_to_post` items, and there is no endpoint to query per-platform prepared content for queued items.

## Requirements

- `/rewrites` GET endpoint must include `ready_to_post` items in result set
- Add read-only endpoint `/prepared-content/{id}` returning per-platform content dict
- Content generation in `/review/{id}/approve` must filter to only platforms where `platform_health.is_healthy = TRUE`
- `remove_from_queue` must clear `verification_status` to empty on unqueue

## Architecture

FastAPI async endpoints with asyncpg queries. The rewrites page query expands to include `ready_to_post` status. A dedicated read-only endpoint provides detailed prepared content for the frontend to display previews. Platform health filtering prevents queuing content for unhealthy platforms.

## Implementation Plan

1. Modify `rewrites_page()` query in `src/web/app.py` to include `ready_to_post` in status filter
2. Add `/prepared-content/{id}` endpoint returning `{"platform_name": "content", ...}` JSON
3. In `approve_review()`, join with `platform_health` to filter verification to only healthy platforms
4. Update `remove_from_queue()` to clear `verification_status = '{}'::jsonb` on unqueue

## Verification Strategy

Link to: `docs/verifications/M13S2V.md` (to be created via `pi verify M13S2`)

## Acceptance Criteria

- `/rewrites` shows items with `ready_to_post` status with distinct "Verified - In Queue" badge
- `/prepared-content/{id}` returns valid JSON with platform-keyed content
- Items with unhealthy platforms cannot be sent to queue (filtered out during approval)
- Unqueue action successfully resets status to `prepared` with empty `verification_status`

## Dependencies

- M13S1 (Database schema with verification_status column)
- M12 (Platform health monitoring system)
