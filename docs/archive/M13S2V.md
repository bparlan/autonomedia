# Verification: M13S2-Backend Query Modifications

## Success Conditions
- `/rewrites` GET returns items with `status = "ready_to_post"` and they are displayed with a distinct "Verified - In Queue" badge.
- `/prepared-content/{id}` returns a JSON object where **each key is a platform name** and the value is a non‑empty string containing the prepared content for that platform.
- When approving a review via `/review/{id}/approve`, only platforms whose `platform_health.is_healthy` flag is `TRUE` are included in the generated content; unhealthy platforms are omitted.
- Executing `remove_from_queue` on a queued item clears the `verification_status` field to an empty JSON object (`"{}"::jsonb`).

## Edge Cases
- **Malformed ID** – Requests to `/prepared-content/invalid-id` should return a `404` with an error message.
- **Missing platform health record** – If a platform has no health row, treat it as unhealthy and exclude it from the approval output.
- **Empty prepared‑content dict** – If no platform has healthy status, the approve endpoint should return a `400` indicating "No healthy platforms available".
- **Large payload** – Verify that the endpoint correctly returns content payloads up to 1 MiB per platform without truncation or memory errors.
- **Concurrent operations** – Simulate two simultaneous `approve_review` calls for the same ID and ensure that only one succeeds while the other receives a `409 Conflict`.

## Failure Conditions
- `/rewrites` omits any `ready_to_post` items.
- `/prepared-content/{id}` returns malformed JSON, missing platform keys, or empty strings for healthy platforms.
- Approve endpoint includes content from an unhealthy platform.
- `remove_from_queue` leaves a non‑empty `verification_status` value.
- Any unhandled exception bubbles up as a `500` without a clear error payload.

## Manual Validation Steps
1. Seed the test DB with a mix of platforms (healthy & unhealthy) and a draft item in `ready_to_post` status.
2. Call `/rewrites` and verify the badge appearance via the JSON `status` field.
3. Request `/prepared-content/{id}` and inspect the JSON shape.
4. Execute `/review/{id}/approve` and confirm only healthy platforms are present in the response and in the DB row `verification_status`.
5. Trigger `remove_from_queue` and assert the `verification_status` column is cleared.

## Regression Risks
- Changes to the `platform_health` join may affect other endpoints that also query platform health (e.g., analytics reports).
- Modifying the `rewrites` query could unintentionally re‑include items that should stay hidden; ensure the filter is scoped only to `ready_to_post` status.
- Clearing `verification_status` on unqueue must not delete other JSON fields that might be added in future specs.
