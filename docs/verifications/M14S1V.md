# Verification: M14S1 - Daily Posting Routine with Randomized Intervals

## Success Conditions
- `posting_routine()` function processes only mastodon-verified `ready_to_post` items (where `verification_status.mastodon.verified = true`)
- Randomized 2-10 minute delays applied between platform batches
- Minimum 1 item posted per execution (if verified content exists)
- Maximum 1-2 items posted per 24-hour window (second item only if 8+ hours have passed since last posting)
- `expires_at` field populated when content is verified (12-hour default TTL, configurable via settings)
- Content prioritized by `verified_at` timestamp descending (recent verified first)
- Manual trigger script `run_daily_posting.py` executes without errors
- Dry-run mode skips browser publishing AND creates no post_history entries (logs intent only)
- Priority-flagged content (`priority: true` in metadata) bypasses expiration for 24 hours

## Edge Cases
- Item with `ready_to_post` status but no `verification_status.mastodon` data should be skipped
- Item with expired `expires_at` (< 12-hour TTL elapsed) should be deprioritized but still processable
- No verified content available → routine logs and exits gracefully without error
- Multiple verified items available → exactly 1-2 items processed based on timing rules
- Content with malformed `verification_status` JSON should be handled gracefully (skip with warning)
- Priority checkbox unchecked → standard 12-hour expiration applies

## Failure Conditions
- `posting_routine()` attempts to post content without `verification_status.mastodon.verified = true`
- Randomized delay outside 2-10 minute range (less than 2 or greater than 10 minutes)
- More than 2 items posted in a 24-hour window
- Dry-run mode creates entries in `post_history` table
- `expires_at` not populated for newly verified content
- Priority-flagged content expires before 24 hours
- Manual trigger script crashes or raises unhandled exception

## Manual Validation Steps
1. Set up test content with `status = 'ready_to_post'` and `verification_status.mastodon.verified = true`
2. Set `verified_at` to current time and `expires_at` to 12 hours later
3. Execute `scripts/checks/run_daily_posting.py --dry-run`
4. Verify logs show intent but no `post_history` entries created
5. Execute `scripts/checks/run_daily_posting.py` without dry-run
6. Verify exactly 1 item posted within 24 hours
7. Wait 8+ hours and re-execute
8. Verify second item can be posted (max 2 per 24 hours)
9. Test expired content (verified_at > 12 hours ago)
10. Verify expired content is deprioritized or skipped
11. Test priority-flagged content
12. Verify priority content bypasses normal expiration

## Test Cases
- **TC1:** Mastodon-verified `ready_to_post` item → posted successfully
- **TC2:** Non-verified `ready_to_post` item → skipped with warning log
- **TC3:** Mixed verified/unverified platforms → only verified platforms processed
- **TC4:** Expired content (verified_at > 12 hours) → deprioritized or skipped
- **TC5:** Priority-flagged content → posts even past expiration window
- **TC6:** Dry-run mode → logs intent, no post_history created
- **TC7:** Randomized delay → measured between 2-10 minutes
- **TC8:** First execution → posts exactly 1 item
- **TC9:** Second execution within 8 hours → no additional items posted
- **TC10:** Execution after 8+ hours → second item may be posted
- **TC11:** Manual trigger script → exits with code 0

## Regression Risks
- Changes to `posting_routine.py` may affect existing `PostingWorker` behavior if module is imported incorrectly
- Database queries for verification status may impact other content listing queries if not properly scoped
- Randomized delay implementation may interfere with test timing if not mockable
- Priority metadata field may conflict with existing metadata structure
